# Nexus 10-Phase Strategic Plan (3-14-26)

## Objectives Overview
1. **Deliver a desktop installable:** Package the unified webapp and Express backend into a native Windows executable.
2. **Be staged for Google Cloud deploy:** Containerize the monolithic structure for scalable serverless hosting.
3. **Backend hardening and additional connectivity:** Secure endpoints and introduce real-time state syncing.
4. **UX and UI Improvements:** Standardize the "War Room" aesthetic and improve application feedback.

---

### Phase 1: Desktop Application Shell (Tauri v2)
**Goal:** Deliver a standalone Windows desktop executable (`.exe`/`.msi`).
- [x] Install `@tauri-apps/cli` and initialize Tauri in the `app` root.
- [x] Configure `tauri.conf.json` to point to the Vite dev server (`localhost:3001` or `5173`) during development.
- [x] Set up the Tauri build pipeline to bundle `dist/` for production.
- [x] (Optional) Compile the Express backend into a standalone binary sidecar to boot headless alongside the Tauri app.
- [x] Build and verify the `.exe` installs cleanly on LuxRig.

### Phase 2: Cloud Run Readiness (Containerization)
**Goal:** Stage the app for Google Cloud deployment for remote Tailscale-free access.
- [x] Write a multi-stage `Dockerfile` in the `app` directory.
- [x] Stage 1: Build the Vite frontend (`npm run build`).
- [x] Stage 2: Copy `dist/` and `server.ts` into a lightweight Node alpine image.
- [x] Expose port `8080` (Cloud Run default) and ensure Express serves static files cleanly.
- [x] Configure a `.dockerignore` to filter out SQLite DBs (`nexus.db*`) and `node_modules`.
- [ ] Perform a local `docker build` and `docker run` to verify functionality.

### Phase 3: Backend Security & Hardening
**Goal:** Lock down the Express backend against brute force and malformed requests.
- [x] Install and configure `helmet` to secure HTTP headers.
- [x] Install `express-rate-limit` and apply it to sensitive `/api/*` endpoints to prevent infinite agent looping.
- [x] Introduce **Zod** schema validation for all incoming POST/PUT JSON bodies (e.g., `/api/brain-link`, `/api/drive/files`).
- [x] Implement a unified error-handling middleware that prevents stack traces from leaking to the frontend.
- [x] Secure the Tailscale network binding so it properly filters non-authorized traffic.

### Phase 4: Google Cloud Console APIs & Optimization
**Goal:** Leverage existing Google Cloud integrations and optimize scaling.
- [x] **Leverage Existing Google OAuth:** Since Google OAuth is already configured (`http://localhost:3001/auth/callback` in `.env`), we will use this native integration combined with Google Identity-Aware Proxy (IAP) or session auth to protect the Cloud Run deployment, rather than introducing Firebase Auth.
- [x] **Secret Manager:** Migrate existing OAuth credentials and Service Account keys from `.env` to Google Secret Manager. Cloud Run will pull these dynamically at boot.
- [x] **Google Cloud Storage (GCS):** Set up a GCS bucket to handle Music Pipeline media assets (wavs, images) alongside the existing Google Drive integration, improving read/write speed for production assets.

### Phase 5: Database Strategy (Local SQLite vs Cloud Postgres)
**Goal:** Reconcile SQLite (local) with Cloud Run (ephemeral).
- [x] Modify `schema.prisma` to support dynamic environment providers.
- [x] Keep `better-sqlite3` for local Tauri runs, ensuring `nexus.db` tracks reliably.
- [x] For Cloud Deployment: Create a **Google Cloud SQL (Postgres)** instance and wire up the Cloud SQL Node.js Connector.
- [x] Set up an automated migration script so that starting the cloud instance auto-migrates the Postgres schema.

### Phase 6: Internal WebSockets Connectivity
**Goal:** True real-time reactivity, abandoning HTTP polling.
- [x] Install `socket.io` (backend) and `socket.io-client` (frontend).
- [x] Wire WebSockets to `server.ts`, allowing it to push state updates directly to the connected clients.
- [x] Refactor the Music Pipeline so that when an LLM agent finishes a background task (e.g., Lyricizing), it emits a `task_complete` event.
- [x] Update frontend UI to listen for these events and refresh widgets without HTTP polling.

### Phase 7: Advanced Agent Tracing (Genkit)
**Goal:** Better observability into LLM prompt chains and agent logic.
- [x] Initialize Google Genkit (`npx genkit init`) within the backend structure.
- [x] Convert the raw `callLocalLLM` fetches into formal Genkit Action/Flow wrappers.
- [x] Wire Genkit's trace exporter to write to local files or a specialized DB so we get full UI visibility of the agent "thoughts".
- [x] Integrate Genkit MCP tool capability so agents can recursively call local functions if needed.

### Phase 8: UX Paradigm Shift (Glassmorphism 2.0)
**Goal:** Premium, consistent aesthetic across all "Front Doors".
- [x] Audit the global `index.css` to confirm all hex codes have been moved to `var(--color-...)` tokens.
- [x] Refactor `PageLayout`, `PageHeader`, and `StatPill` globally so that every single page adheres to the new design system.
- [x] Standardize background blurs (glassmorphism), hover states, and dynamic micro-animations.
- [x] Verify that there are zero centering/layout conflicts across the Labs, Music, and Video Studio pages.

### Phase 9: Pipeline Visual Flow UI
**Goal:** Interactive control over the Music and Data pipelines.
- [x] Implement `reactflow` on the Pipeline page to replace the text-based list.
- [x] Create custom Node components for "Idea", "Lyrics", "Suno", "Video", and "Publish".
- [x] Connect the visual nodes to the underlying database states so progress changes visually in real-time.
- [x] Add drag-and-drop or right-click context menus for manual override (e.g., "Force Regenerate Lyrics").

### Phase 10: Final Deployment & Sync (Nexus 2.0)
**Goal:** Ship it.
- [x] **Deployment 1 (Local):** Verify the `Tauri` desktop app works flawlessly on LuxRig, hitting the local SQLite DB and local Ollama model.
- [x] **Deployment 2 (Cloud):** Push the Docker container to Google Cloud Run, hitting the Cloud SQL DB and Google Drive API.
- [x] Write a "State Sync" manual utility script that backs up local `nexus.db` states to the cloud DB, so your desktop and remote ops stay loosely connected.
- [x] Freeze the codebase and tag version `v2.0.0`.
